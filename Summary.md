## 个人使用经验

> 我个人更贴近于Token价值论，即想办法去提升**Token的使用价值**，而不是去一味追求尽量少的Token消耗。

- 现阶段的AI更像是一个**聪明的傻子**，它的知识储备量已经远超人类，专业程度上也是要超过绝大部分的专业从业者的，但它的思考能力和判断能力依然是有限的。并且不具备完善的**前瞻能力**。一个正常开发者有的时候会在设计某些类的时候下意识的添加一部分设计, 用来准备为后续的功能扩展而做一些准备。但是AI是不存在这种前瞻性的设计的，它只会根据你当前的需求去生成代码，而不会考虑未来可能的需求变化。(类似于有时候你会写todo等)

```cpp
class Component {
    /* logic code here...
    void methodA() {}
    void methodB() {}
    void methodC() {}
     */
    /* sometimes devs will add some more methods like this
    some will be used in the future
    */
};
```

- AI对于我而言，更多的是一个**辅助工具**，是的仅仅达到**工具的层级**，而不是一个**合作开发者**。它可以帮你完成一些重复性工作，或者帮你快速生成一些代码片段，但它不能完全替代你的思考和判断能力。尽管他可以快速完成一些复杂问题（例如: LeetCode的困难题目，让他帮你实现某些算法等复杂操作）
- 本质上他依然是开发者为了减轻心智负担的一个方法，他所处的层级会更像是编程语言对于汇编语言的关系，或者是IDE对于文本编辑器的关系。
- 就目前而言，我个人认为公开的所有AI几乎都可以满足大部分的开发需求，主要区别在于**使用体验**和**使用成本**的二者平衡的问题。即：我很有钱，不会去考虑Token燃烧的问题，我只希望我几轮对话就能得到完整成品。还是更希望低成本的开发，同时兼顾到一定的使用体验。
  如果是前者，我建议是无脑选择头部的AI平台（如openai，Anthropic,GLM等）会显著帮你减轻开发负担，甚至可以直接帮你完成大部分的开发工作。
  如果是后者，则各大AI平台均可以作为你的选择之一。是的，所有平台均可以满足大部分的个人小项目的开发需求。~~免费的平台请慎重，豆\*等~~

# 常见问题

说到AI辅助开发，我们绕不出一些问题就是 contextWindow(上下文窗口大小)，SKILL，Prompt，Memory等，还有更加关键的隐性问题：AI自身幻觉率，AI降智，以及MoE架构可能会导致回答不一致性的问题。对于个人主观问题而言则是：为啥AI插手一段时间后，我自己没法改自己的代码了（没有办法工程化）。

## 适当控制上下文窗口大小

- 尽管现在很多模型的上下文窗口已经扩大到了1M token，看似很大，给人一种AI更加聪明的错觉，但是就个人经验而言，这个窗口过大反而会导致一些不必要的问题出现，AI输出结果是根据输入的上下文进行推理的，如果上下文过大(你的对话历史过长)，AI会因为对话中存在大量重复性、分散的信息而降低处理能力，并且有些AI也会出现无法充分利用上下文内容（专业部分请参考：Lost In the Middle）。就如同当你阅读到这里的时候，你还记得我在开头问了几个问题吗？每个问题有几种选择？AI同样也会出现这种问题。

这就跟你给队友报点，两句话就完事，你非要来一段报菜名，你看看你队友骂不骂你就完事了

**解决方案**：根据问题或者文件拆分，根据难度重新开一轮新的对话

这个方法就像是给人一份说明书去组装一个乐高一样，不断通过说明书去强化下一步需要怎么去做，而不是一辆车最后组装成擎天柱。

- 问题简单（比如修改接口，调整参数，添加部分函数，简单函数或者模块修改），可以只用一轮对话解决。
- 问题复杂（比如涉及系统设计、架构调整、实现复杂系统功能等），可以拆分多轮对话进行解决，如果不能解决，让他生成一个Plan并落地到本地文件，然后让AI每完成一部分就停止，然后人再去审核代码、修改Plan。并在下一轮对话中通过强制加载的方法来给AI进行一次对话的强制刷新Plan。

## SKILL，Prompt，Memory （三者一起说） 的作用

> 从 客观意义（系统工程角度）来说，Skill、Prompt、Memory 本质上**不是同一种东西**，但它们都属于"**给模型提供额外上下文（Context）**" 的一种形式。
> 如果你想知道的更详细，可以参考：https://agentskills.io/specification

SKILL在早期的需求似乎更加贴近**技能**补全。但最近我更觉得SKILL的职能已经不再局限于技能补全，更加贴近于**规范化**限定。比如:

> - 告诉它仓库结构、命令、约定
> - 固定安全边界与权限
> - 规定团队风格、架构选择、验收标准
> - 把重复流程变成稳定工作流

举个例子：
AI的“知识库”十分巨大，你提出问题，实际上就已经开始对这个“知识库”进行限定了
```
优先在软件开发领域寻找答案
语言限定为：xxxx
开发框架为：xxxx
目标平台为：xxxx
```
SKILL的作用则是帮你把这个限制精度进一步提高。


生活化一点的例子就是

> 你去花店给ta买花。
> - 如果ta只是说“我喜欢花”，你想想自己的选择难度。
> - 但是如果ta说“我喜欢郁金香，色调为粉色，20支，5种搭配，拍照用/生活装饰用”，你就知道该怎么买什么花了。
> 其实AI也是同理~

## AI自身幻觉率，AI降智，以及MoE架构可能会导致回答不一致性的问题

幻觉是不可控的，不同平台测试的结果差距也比较大，AI降智对于用户而言也是未知的，MoE架构可能会导致回答不一致性的问题，这个问题在多轮对话中尤为明显。同一问题多次回答可能不一致，其原因可能包括生成采样、上下文差异、推理实现以及服务端模型版本变化。仅凭用户侧表现通常无法判断具体原因。

缓解这个的措施：当你感受到AI的回答出现不太正常的情况时，临时换一个模型去回答同样的问题（这也是为什么我个人的开发环境中会倾向同时存在多个不同平台AI），或者单开一个新的对话窗口（彻底清空上下文）更换问题问法，然后去分析和对比不同模型的回答，然后再去综合分析，最终得出一个相对合理的结论。

## 工程化问题

这是一个非常重要的问题，尤其是对于个人开发者而言。如果完全交由AI进行，则会进入到野蛮开发的状态。最终的结果就是：
AI用了一堆基础元素给你搭建了一个炫酷的UI，然后都给你塞进一个index.html（经典html笑话）。或者就是一个功能都被整合进一个class中，这种情况就不是一杯茶、一包烟一段代码改半天了。相信你的Token消耗也会很美丽
但实际开发环境下，更多的是需要考虑模块边界问题，在某些特殊方向上你甚至需要考虑模块内存大小等更加深层的问题。AI在这方面的能力是有限的，尤其是对于个人开发者而言，AI无法完全理解你的设计意图和代码结构。

## 额外SKILL安装

全局安装SKILL建议只存放某些通用性、格式化、或者测试流程SKILL.

个人不太推崇过多SKILL到一个项目中。有以下几个原因：

1. SKILL**安全性**问题。毕竟不管是codex、copilot或者其他的插件，在日常场景下客户端收到用户的授权是很高的,详细问题请参考：提示词（或SKILL）注入攻击
2. SKILL的使用**成本问题**。每次调用SKILL都会消耗一定的Token，如果你在一个项目中安装了过多的SKILL，那么在开发过程中，AI每次调用SKILL都会消耗大量的Token，从而导致开发成本的增加。

但是我也不太推崇无SKILL的开发方式。

1. **有些库目前还处于活跃开发期**，比如shadcn,尽管AI可以通过联网搜索到相关的文档，但是权威性依然不如他们官方自己出的文档。并且有可能会出现污染性内容出现（就比如我这篇文章）。
2. **小众的库**，有很多小众的第三方库，网上也没有太多的资料，AI也不一定能准确的理解这些库的使用方法和原理。这个时候如果你有一个专门的SKILL去帮助AI理解这些库，那么就可以大大提高开发效率。
3. **无相关语言/方向开发经验的开发者**，是的，我说的就是我。真的不太喜欢写前端，TypeScript只浅浅了解了一周就开始上手了。这个时候真的需要一些比较完整的SKILL去帮助上手。就当做是为了省时间交的学费吧
4. **完全无开发经验的开发者**，请拿SKILL当做学骑自行车的那个辅助轮。这个应该是你当前做的项目的最后一道保险了。
5. **坚持依靠古法编程的朋友**，安装一些特性SKILL，用来减少debug的时间。
