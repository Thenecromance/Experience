# SKILL、Memory、Prompt

> 这部分大概率对于非专业开发者而言能接触到的除了对话以外最多的部分

接下来讲解会将这3者整体简化为**单纯的文字**，至于其背后的原理机制，类似**Attention Mask**、**Embedding**、**Tokenization**等不在讨论范围中

只需要简单的记住：这3者都会随着你的对话按照某些规则进行上传，并且其**权重**在某些模型中会被放大，并最终比较显著的影响结果。

---

## SKILL是否真的有必要？

> **Skill 实际上是在不断缩小搜索空间（Search Space）**

部分开发者可能会认为SKILL、Memory、Prompt可以解决所有问题，或是神话化其作用，但还有一部分开发者可能会认为SKILL、Memory、Prompt是鸡肋，甚至是无用的

实际上，这个方面的设置更多是取决于**个人需求和项目场景**。
现在假设如下几个场景：

- **一个非专业开发者**（很少或是从未接触过软件开发的人）
  由于其从未接触过如何设计一个系统，SKILL在这个场景下则是非常有用的，甚至是必不可少的。SKILL在这里的作用是非常明确的：**帮助你去缩小AI需要寻找的知识范围**从而帮你避免由于你很难**表述准确**而导致AI可能会从不同专业方向去生成答案。

  就拿UI（并非UI设计，是的这个是完全不同的两个方向）来举例，在C++开发中，如果你只是简单的说

  > 协助我搭建一个UI
  - AI生成的结果可能会是使用Imgui,
  - 也有可能是给你搭建一个QT的UI，
  - 甚至是使用其他的UI框架。

  **但实际上，你可能只是想用ImGui在窗口内创建一个UI。**

  此时SKILL的作用就是协助你去缩小AI的知识范围，你的回答在AI眼里可能就会类似于：

  > “你是一个C++开发者（语言范围限制），你帮我在Windows平台上的OenGL(平台限制)中，使用成熟的第三方库（进一步限制），**_协助我搭建一个UI_**”

- **你是一个普通开发者**
  由于你已经具有一定开发经验，至少知道如何比较精确的表述你的问题。
  基础性SKILL此时可能会变成你对话中的一种**污染**，甚至是干扰
  但是有些SKILL仍然是**有用**的，类似某一些处于**活跃开发阶段的第三方库**。

  > 众所周知，第三方库的更新迭代是非常快的，甚至有些库在半年内就会有一次大的更新迭代。
  > 然而，AI的知识库并非是实时更新的，甚至有些库的知识库可能会滞后半年以上。
  > 尽管AI依然可以通过**搜索引擎**去获取最新的API文档，但搜索引擎的结果可能会将某些**不相关的内容**也一并返回给自身（就比如说可能会引用C\*N,甚至本篇文章），那么AI生成的结果要不就是错误的，要不然就会不断的去消耗你的Token去尝试获取正确的结果。

  这个时候，SKILL的作用就变成**提升Token质量**

- **在这个方向上，你是一个专业开发者**
  这个阶段，SKILL似乎对你来说就是完全可有可无的了，甚至是鸡肋。SKILL的唯一作用可能就是帮你节省掉你详细描述问题的时间。
  消耗的Token会更多，即Token与省事之间的平衡问题。

**这个场景就更像是**：

- 你是一个新手厨师，你要做一道菜，那么**说明书**(SKILL)就是你必备的。
- 但是如果你是一个比较有经验的厨师了，一些**基础菜系**（基础性SKILL）就不再那么必要，但是一些**新菜系**（第三方库的SKILL）就变得非常有必要了。
- 但如果你是一个专业做川菜的厨师，并且你只需要做川菜，那么看着说明书（SKILL）的做菜似乎就很多余了。

## 准确表述

> **_哭？哭也算Token的哦_**

对于AI来说，**主观的修饰词**几乎都是污染。

假设你的 Skill 写的是：

> UI 要现代、美观、好看。

这属于低约束

AI 可以生成：

- 苹果风
- Material Design
- Cyberpunk
- 极简
- 拟物
- 渐变风
- 玻璃拟态

因为"好看"没有客观定义。

---

如果 Skill 写成：

> 整体采用 Apple Human Interface Guidelines 风格。
> 大量留白。
> 8pt Spacing。
> Border Radius 16px。
> 阴影仅使用弱阴影。
> Primary Color #007AFF。
> 字体 SF Pro，若不可用则 Inter。
> 禁止使用 Material 风格按钮。
> 不允许使用高饱和颜色。
> 不允许超过三级视觉层级。

那么模型几乎没有什么自由发挥空间。
生成结果会稳定得多。

## Skill 最好的写法其实不是"描述"

> AI不是人，本质是没有人类情感的。情感词，重复表达**几乎**都是噪音或是污染。
> 哪怕是人，在没有限定的条件下也是如此。
> e.g. 现在你想想:有质感, 大气。 这种词汇，你是否能准确的描述出你想要的东西呢？甚至你自己都不清楚你想要的是什么。

很多人写 Skill：

```markdown
请设计现代化UI。
要求美观。
要求高级。
要求简洁。
```

对于AI而言，这些都是没有任何约束能力。

然而真正好的Skill写法应该更加贴合一个十分严谨的**设计文档**，甚至是一个**设计规范**

```markdown
Layout

- 12 Column Grid
- Max Width 1280px
- Gap 24px

Spacing

- 8pt System
- Outer Margin 32px

Typography

- H1 32 Bold
- H2 24 SemiBold
- Body 16 Regular

Card

- Radius 16px
- Shadow 0 2 8 rgba(...)
- Padding 24

Button

- Radius 999px
- Height 44px

Interaction

- Hover Scale 1.02
- Transition 200ms ease

禁止：

- 彩虹渐变
- Glassmorphism
- Neon
- 超过三级阴影
```

这已经不像 Prompt。
更像设计规范（Design System）。
LLM **非常擅长**遵循这种规则。

## Token 会不会很多？

> 即Token价值论，你需要考虑的是如何**最大化Token的价值**，而不是如何**最小化Token的消耗**

很多人担心：

> Skill 越详细 Token 越浪费?

其实不是。
如果这些 Token：

- 消除了歧义
- 避免反复修改
- 提高一次成功率

那么这些 Token 的价值非常高。
真正浪费 Token 的往往是：

```
请再改一下

还是不太对

再高级一点

还是不像苹果

颜色不舒服

字体再换一下

...
```

这种反复迭代消耗的 Token，通常远大于一开始提供一份清晰规范。

## 需要注意的一个平衡点

> 请“惜字如金”，不是银，不是铜，更不是铁。是金

就如同上面这段文字，其中对于AI有效的Token只有：

- **惜字如金**具有价值
- **不是银，不是铜，更不是铁。是金**则属于重复限制，应用中，这里只会产生负面效果

如果 Skill 包含大量**几乎永远不会影响输出的细节**，就会产生收益递减。例如：

- 每个组件都指定像素级坐标（除非必须复刻）
- 写几十条很少触发的例外规则
- 重复表达同一约束
- 把每种组件的所有可能状态都展开，而实际上项目只会用到少数几种

这会增加上下文长度，却未必带来明显提升。
因此，一个优秀的 Skill 应该做到：

- 高约束：明确风格、布局、交互原则，减少模型自由发挥。
- 低歧义：避免使用“高级”“现代”“好看”等主观词，尽量用可执行、可验证的规范描述。
- 高复用：把所有页面都遵循的规则放进 Skill，把页面特有需求留给当前 Prompt。
- 避免冗余：每条规则都应能影响最终输出，而不是重复或极少生效。

对于UI 设计风格这种一致性要求极高的场景，一份高质量的 Skill 往往更接近一份 Design System + Design Specification（设计系统 + 设计规范），而不是一篇自然语言说明。这样的 Skill 通常能获得最稳定、最一致的生成效果。
